The Challenges

Why LLM Agents Fail at Coding

LLMs can write impressive blocks of code, but without structured constraints they fall victim to four recurring pitfalls that slow down development.

😤

Wrong Assumptions

Models silently choose an interpretation of an ambiguous prompt and run with it, hiding confusion rather than asking clarification questions or presenting trade-offs.

🔧

Speculative Bloat

A tendency to overcomplicate APIs, design overly abstract classes, and write 1,000 lines of "future-proof" boilerplate when 100 lines of simple logic would suffice.

⏱️

Collateral Damage

Unintentional side-effects where LLMs rewrite, reformat, or delete adjacent code, docstrings, and comments that they didn't fully comprehend.

🧪

Silent Failures

Finishing a requested feature and declaring it complete without executing the test suite, verifying types, or ensuring a build actually compiles.

Karpathy Guidelines Solve This.

By defining explicit constraints, these guidelines force the AI agent to verify assumptions, prioritize absolute simplicity, limit file edits surgically, and prove correctness before finishing.

Four Pillars

The Core Principles

Four rules structured to guide LLMs to think clearly, keep code compact, minimize edits, and verify everything.

🎯

Think Before Coding

State assumptions explicitly before writing code. If multiple interpretations exist, present them rather than deciding silently. Push back on overengineered requests and stop to clarify any ambiguities immediately.

Simplicity First

Implement the absolute minimum code necessary to solve the problem. No speculative features, Single-use code must not be wrapped in abstract helper modules. If a 200-line solution can be written in 50 lines, rewrite it.

🔍

Surgical Changes

Only touch files and code blocks that are essential to the task. Avoid "cleaning up" adjacent logic, comments, or formatting. Maintain styling consistency exactly, even if you disagree with the local architecture.

04
🤖

Goal-Driven Success

Anchor agent success to tests first. Always write tests or describe exact verifiable success criteria. Verify compilation, run typecheck commands, and run relevant test suites to ensure zero regressions before completing a task.

Quickstart

How to Integrate

You can apply these rules directly to Claude Code, Cursor, or Agentic MCP-compatible setups.

01

Claude Code (CLAUDE.md)

Place a CLAUDE.md file in the root of your project directory. Claude Code will automatically detect and consume it as system instructions.

# CLAUDE.md - Behavioral guidelines to reduce LLM mistakes
## 1. Think Before Coding
- State assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them - don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop and ask.

## 2. Simplicity First
- No features beyond what was asked.
- No abstractions for single-use code.
- No speculative flexibility or configurability.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

## 3. Surgical Changes
- Touch only what you must. Clean up only your own mess.
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style perfectly.

## 4. Goal-Driven Execution
- Define success criteria upfront. Write tests first where possible.
- Run tests and lint checks to verify success.
- Do not stop until all criteria are met and verified.

Cursor Rules (.mdc)

Add karpathy-guidelines.mdc to your project's .cursor/rules/ directory to apply them automatically to Chat and Composer.

---
description: Behavioral guidelines derived from Karpathy's LLM coding observations
globs: *
alwaysApply: true
---
# Karpathy Guidelines

Apply the following 4 behaviors strictly:
1. **Think Before Coding**: Ask questions for ambiguities, state assumptions first.
2. **Simplicity First**: Write minimal code. No speculative abstractions.
3. **Surgical Changes**: Only edit lines critical to the task. Maintain matching style.
4. **Goal-Driven Execution**: Run compilation checks and tests to verify success.
